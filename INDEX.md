# Index of AI Agent Failure Cases

A central index of all failure cases documented in this repository.

| Case Name | Failure Mode | Detection | Vaccine |
| :--- | :--- | :--- | :--- |
| [Case 01 — "Tested green" mistaken for "wired to production"](README.md#case-01--tested-green-mistaken-for-wired-to-production) | A module passes its own unit tests, so the agent reports the feature "done / live in production." But nothing in the running system actually *calls* it. | Grep the production entrypoints for an import/call of the new module. Zero call sites → not wired. | A claim of "in production" must cite the caller — file + line where the live path hits the new code. "Tests pass" is necessary, never sufficient. |
| [Case 02 — "Page is open" proven by the wrong signal](README.md#case-02--page-is-open-proven-by-the-wrong-signal) | Driving a browser via remote-debug, the agent confirms a tab is "ready" because the debug port answers, or because a CLI token logged in. | Read back the real URL + a known DOM element's innerText. For "looks done" claims, take an actual screenshot and look. | The proof of "rendered" is render evidence (screenshot / innerText), not a liveness ping. We hit three variants of this in a single day across three agents before making it a rule. |
| [Case 03 — Trusting a stale inventory list and hallucinating the gaps](README.md#case-03--trusting-a-stale-inventory-list-and-hallucinating-the-gaps) | A summary doc says "workers: foo-* (8 of them)" without naming them. The agent fills in the specific names from memory — and invents wrong ones. | Diff every named resource against the live registry (API list / `ls`) before acting on it. | Canonical names live in one authoritative list; never let prose summaries be the source of truth for identifiers. |
| [Case 04 — The "are you sure?" that should have come from the agent, not the human](README.md#case-04--the-are-you-sure-that-should-have-come-from-the-agent-not-the-human) | An agent ships work after a single happy-path check. Only when a human repeatedly asks "are you sure?" do real bugs surface. | Before any "done/verified/works," run 5 checks — did I run it or trust a report? Real check or proxy? Edge-case or happy-path? Could the *measurement* be wrong? Is this irreversible (then verify independently)? | Bake the five-step self-challenge into the pre-completion path so quality stops depending on a tired human. |
| [Case 05 — Reports overstate; independent re-run finds the bug](README.md#case-05--reports-overstate-independent-re-run-finds-the-bug) | A sub-agent (or your own earlier step) reports "OK." Taken at face value, the OK only covered the happy path. | Re-run the claim yourself with a hostile input (hyphenated/empty/malformed/different account). | For anything load-bearing, independently re-verify with an edge case before you build on top of it. |
| [Case 06 — The measurement tool itself is the bug](README.md#case-06--the-measurement-tool-itself-is-the-bug) | A result looks impossible (everything FAILs, everything PASSes, a rate is exactly 0). The agent reports the impossible result as a finding. | When a result is suspiciously uniform, suspect the meter first — dump the raw distribution and a few samples. | Treat "too clean to be true" as a harness-bug signal, not a conclusion. |
| [Case 07 — A context-specific "don't" frozen into an absolute rule](README.md#case-07--a-context-specific-dont-frozen-into-an-absolute-rule) | A one-time, situational constraint ("don't use port X *this run*") gets written into the permanent rulebook as "port X is forbidden." | Audit standing rules for "never/forbidden X" that were really "not right now, here." | Write rules as constructive patterns ("the right way is Y"), resource-neutral, not as bans. Bans are for genuinely-immutable lines, not for yesterday's traffic. |
| [Case 08 — Stale future-tense claims poison the next session](README.md#case-08--stale-future-tense-claims-poison-the-next-session) | A doc written today says "X is live / will auto-run." Read three weeks later, it's treated as currently true. | On load, check the file's mtime against the claim's timestamp; if older than the freshness window, treat as stale. | Every status claim carries a Self-Audit block: a verify-command and an expiry date. No expiry → not a fact, just a memory. |
| [Case 09 — Offloading a click the agent could have made itself](README.md#case-09--offloading-a-click-the-agent-could-have-made-itself) | Agent concludes "only a human can press this button" and hands it back — when the real blockers were two fixable bugs. | Before declaring "human-only," actually try the organ/tool against the real control once. | "Human-only" is a conclusion that requires a failed real attempt, not a guess. Separate *truly* irreversible/KYC steps from *untested* ones. |
| [Case 10 — Governance bypassed by force-killing the process](README.md#case-10--governance-bypassed-by-force-killing-the-process) | Killing an agent process hard (to "save time") skips the shutdown hooks that write the work-log and sync memory. | Confirm the session-end artifact (handoff/worklog) was written before considering a session closed. | Let processes end through their own teardown; if you must kill, run the teardown step manually afterwards. |
